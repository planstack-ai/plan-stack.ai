コンテキスト設計の原則

「以前はすべてをコンテキストに詰め込んでいました。今は慎重に選んでいます。」

そのマインドセットの転換—「何を渡せるか」から「何を渡すべきか」へ—が、 平凡な結果と優れた結果の違いです。コンテキスト設計とは、含めるすべての情報に対して 意図的であることです。

核心の問い

何かをコンテキストに追加する前に、自分に問いかけてください： 「Claude がこの特定のタスクを完了するためにこれが必要か？」

「いつか役に立つかも」や「これは関連がありそう」ではありません。 今、この正確なタスクのために Claude がそれを必要としているか？ 答えが明確な「はい」でなければ、除外してください。

判断フレームワーク

含める可能性のあるすべてを3つのバケットに分類してください：

必須

直接必要。これがなければ Claude はタスクを完了できない。常に含める。

役立つ

有用なコンテキストを提供。精度を向上させる可能性がある。余裕があれば含める。

ノイズ

タスクと無関係。気を散らすだけ。絶対に含めない。

実践的なガイドライン

  1. 最小限から始めて、必要に応じて追加。 必須のコンテキストだけから始める。Claude が質問したり不完全な結果を出したりしたら、追加する。 すべてを最初から入れない。
  2. 大きなファイルは要約する。 2,000行のファイルを貼り付ける代わりに、構造を説明する：「このファイルには認証メソッドを持つ User モデルが含まれています。関連する関数は145行目の authenticate() です。」
  3. すべてではなく構造を渡す。 バグ修正には、壊れた関数とその直接の依存関係を渡す—コードベース全体ではなく。 新機能には、インターフェース定義と類似機能の例を1つ渡す。
  4. 関心を分離する。 認証に取り組んでいる？課金コードを含めない。API に取り組んでいる？ フロントエンドコンポーネントを含めない。スコープを厳密に。

目標は Claude が必要かもしれないすべてを与えることではありません。 目標は Claude が必要とするものを正確に—それ以上でもそれ以下でもなく—与えることです。

チェックリスト

プロンプトを送信する前に、これを確認してください：

  • 含めたすべてがこのタスクに必須か？
  • 削除できる「念のため」のものはないか？
  • 最も重要な情報を最初に置いたか？
  • 同じ情報をより短く伝える方法はないか？

重要なポイント

  • 「何を含められるか」ではなく「Claude が何を必要としているか」を問う
  • コンテキストを必須、役立つ、ノイズに分類する
  • 最小限から始め、結果が不完全な場合のみ追加
  • すべてを貼り付けるのではなく大きなファイルを要約する
