コンテキストウィンドウとは

よくある話です。Claude Code にコードベース全体を渡します。すべてのファイル、すべてのフォルダ。 20万トークンのコンテキストがあるのだから、全部使わない理由があるでしょうか？ すると Claude は的外れなものを生成します。アーキテクチャを無視する。 既存のコードを複製する。何が間違っていたのでしょうか？

誤解はシンプルです：長いコンテキスト = 常に正確。 これは間違いです。そしてその理由を理解することが、Claude を効果的に使う第一歩です。

デスクのアナロジー

コンテキストウィンドウはファイリングキャビネットではなく、デスクと考える

ファイリングキャビネットは書類を保管します—何千ページも詰め込めます。 デスクは実際に作業する場所です。紙を広げすぎると、何も見つけられなくなり、 明確に考えることも不可能になります。コンテキストウィンドウはあなたのデスクです。 ストレージではなく、作業メモリなのです。

トークンとは？

トークンは Claude がテキストを測定するために使う単位です。おおよそ：

  • 1トークン ≈ 英語で4文字
  • 1トークン ≈ 日本語で1〜2文字
  • 典型的なコードファイル（200行）≈ 1,000〜2,000トークン

Claude が「20万トークンのコンテキストウィンドウ」を持つとき、それは一度に約 15万語を保持できることを意味します。本約500ページ分、 または100〜200個のコードファイルに相当します。

「読める」vs「注意を払える」

重要な洞察はこれです：Claude は20万トークンを読めますが、 そのすべてに等しく注意を払うことはできません。

500ページの本のすべてに一度に集中できないのと同じように、 Claude の注意力には限界があります。長いコンテキストの中間にある情報は 体系的に無視されます—「Lost in the Middle」と呼ばれる現象です。

つまり、すべてをコンテキストに詰め込んでも助けにならない。むしろ悪影響があります。 ノイズが増えればシグナルは減る。モデルは重要なものを見つけるためにより懸命に働かなければならず、 しばしば失敗します。

本当の問い

「どれだけ入れられるか？」ではなく、「この特定のタスクのために Claude が本当に知る必要があることは何か？」と問い始めましょう。

そのマインドセット転換—入力の最大化から入力の最適化へ—が、 イライラする Claude セッションと生産的なセッションを分けるものです。

重要なポイント

  • コンテキストウィンドウはストレージではなく作業メモリ
  • 20万トークン ≠ 20万トークン分の質の高い注意
  • 「読める」と「注意を払える」は別物
  • コンテキストが多いほど結果が悪くなることが多い
