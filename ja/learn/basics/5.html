実践的なワークフロー

理論は素晴らしい。でも実際に日々どう使うのでしょうか？ ここにすべてをまとめた具体的なワークフローがあります。

ステップバイステップのワークフロー

明確でスコープされたタスクを定義する

何かを入力する前に、何が欲しいか正確に把握する。「ログインバグを直して」は曖昧。「メールが空のときに authenticate() で発生する null ポインタ例外を修正」はスコープされている。

最小限のコンテキストを集める

Claude が絶対に必要とするものを特定する。バグ修正の場合：壊れた関数、エラーメッセージ、直接の依存関係。それ以上は不要。

プロンプトを構造化する

タスクの説明を最初に置く。次に必須のコード。最後に制約や要件。覚えておく：最初と最後が最も注意を引く。

レビューして反復する

結果が不完全なら、段階的にコンテキストを追加する。間違っているなら、タスクを明確にする。ただコードを追加して上手くいくことを祈らない。

CLAUDE.md のセットアップ

プロジェクトのルートに CLAUDE.md ファイルを作成してください。 このファイルはすべての会話の最初に自動的に読み込まれ、永続的なコンテキストを提供します。

含めるもの：

  • プロジェクト固有の規約（命名、パターン）
  • すべてのタスクに影響する技術スタックの詳細
  • 「落とし穴」や明らかでない制約
  • 類似の実装を見つける場所

CLAUDE.md は短く保つ—200行以内。すべてのメッセージで消費されるので、 ここが膨れるとすべてのやり取りに影響します。

よくあるシナリオ

バグ修正：壊れた関数、エラーメッセージ、1レベルの依存関係を渡す。バグが必要としない限り、ファイル全体を渡さない。

新機能：インターフェース定義、類似機能の例を1つ、アーキテクチャの制約を渡す。必要なら Claude に追加を求めさせる。

リファクタリング：リファクタリングするコードを渡し、目標状態を説明する。望むパターンの例が他にあれば1つ含める。

シリーズの振り返り

基礎で学んだこと：

  1. コンテキストウィンドウ — ストレージではなく作業メモリ
  2. トークン消費 — 履歴は蓄積し、出力もカウントされる
  3. 精度と長さ — 中間は忘れられ、短い方が良い
  4. コンテキスト設計 — 必須 vs 役立つ vs ノイズ
  5. 実践的なワークフロー — スコープ、最小化、構造化、反復

重要なポイント

  • 明確でスコープされたタスク定義から始める
  • 必須のコンテキストだけを含める；必要な場合のみ追加
  • プロンプトを構造化：タスクが最初、コードが次、制約は最後
  • 永続的なプロジェクトコンテキストのために CLAUDE.md をセットアップ

これで Claude Code を効果的に使う準備ができました。
